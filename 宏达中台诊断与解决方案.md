# 三、AI Provider层详解

## 加载流程
getProvider() → loadProviderConfig() → AI_PROVIDER=deepseek → 创建DeepSeekProvider → 检查DEEPSEEK_API_KEY → 缺失则fallbackToMock

## DeepSeekProvider (lib/ai/providers/deepseek.ts)
- 使用OpenAI SDK, baseURL=https://api.deepseek.com
- 调用client.chat.completions.create()
- 支持response_format: { type: 'json_object' }
- ***关键问题: 没有设置请求超时(timeout)***
- 失败时降级到MockProvider的固定默认输出

## MockProvider (lib/ai/providers/mock.ts)
- 不依赖任何API key
- 所有脚本相关请求返回同一段固定文案: "客户只发一张图，问我多少钱..."
- ***完全不区分账号、材质、痛点等上下文***

# 五、核心问题分析 (共8个)

## 问题1: DeepSeek在Vercel上全线超时(***根本原因***)
现象: 所有调用DeepSeek的API端点全部降级到Mock或规则引擎
证据: API返回mock:true, 内容固定
根因:
- DeepSeek API在Vercel上超时(超过10s Serverless限制)
- DeepSeek SDK未设置请求超时
- 降级到MockProvider后返回固定写死内容
- 规则引擎模板内容与用户输入的用户名、账号人设、具体痛点不相关

## 问题2: MockProvider返回固定内容
mock.ts的getMockResponse()只检查prompt关键词,不解析上下文
```javascript
if (prompt.includes('脚本'))
  return JSON.stringify({ hook: '客户只发一张图，多少钱？我不敢直接报。'...});
```

## 问题3: 客户端兜底硬编码(未部署)
钩子fallback → 5条固定钩子(刚改为动态生成但未部署)
角度fallback → 8条硬编码角度(已改为本地生成但只用输入关键词拼接)

## 问题4: runPipeline规则引擎模板与账号/知识无关
- 模板12种,内容固定为"热转印"场景
- 不区分账号人设(小林/小陈/老板/沐森兄)
- 不引用知识卡内容

## 问题5: script-pipeline的generateHook随机取
从写死的HOOK_FRAGMENTS数组中随机取一条,与用户输入无关

## 问题6: 环境变量缺失
.env.local没有AI_PROVIDER和DEEPSEEK_API_KEY
本地开发默认使用MockProvider,无法测试DeepSeek连接

## 问题7: 多个API端点并存导致调用链混乱
6个API端点: pipeline, hooks, angles, generate-script-draft, generate-script-pipeline, generate-script-strategy
前端handleGenerate()调用pipeline API但不传入钩子和角度
最终生成脚本从来没用过用户选择的钩子和角度

## 问题8: 代码部署流程阻塞
沙箱无法运行git命令,每次修改需要用户手动推送

# 六、优先修复方案

## P0: DeepSeek超时/不可用
方案A1: DeepSeek SDK加5s请求超时 timeout: 5000
方案A2: 换OpenAI AI_PROVIDER=openai + 有效API key

## P0: Mock Provider固定内容
方案B2: 规则引擎模板使用{material}, {pain}, {accountName}变量

## P1: 钩子/角度/生成与输入无关
方案B2 + B3: 角度和钩子完全本地化,使用模板+输入拼接

## P1: 前端handleGenerate不使用选择的钩子和角度
方案C1: pipeline API传入所有参数(hook/angle/account/strategy)

## P2: 多个重叠API端点
方案C3: 清理到2个入口(script/pipeline + script/batch)

## P2: 部署流程不畅
方案D1: GitHub Actions自动部署(push→vercel deploy)
